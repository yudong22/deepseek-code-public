# v0.6.0 实施计划：Coding 能力 + 治理与可信 (合并优化版)

本方案作为 `v0.6.0` 的完整设计与实施计划，融合了 **Claude Code (CC) 方案的完整业务范围** 与 **Antigravity (AG) 方案的基建与安全硬化思路**。我们通过折中的“同步 Trait + 轮询取消/超时”设计，在避免大面积重构的前提下，实现高水准的安全护栏与新工具对齐。

---

## 关键设计决策 (合并共识)

1. **Tool::execute 保持同步，通过 `cancel_flag` 协作取消**
   - **决策**：不修改 `Tool` trait 签名，不引入 `async-trait` 依赖。保持现有 102 个测试和 7 个工具的同步结构。
   - **取消机制**：在 `ToolContext` 中引入 `cancel_flag: Arc<AtomicBool>`。
   - **Bash 超时与取消**：`BashTool` 内部通过 `std::process::Command` 启动 `child`，并在 `spawn_blocking` 线程中通过 `child.try_wait()` 循环（每 50ms 睡眠）进行非阻塞等待，轮询检查 `cancel_flag` 和 `timeout`。一旦触发，直接调用 `child.kill()`，**彻底解决子进程泄露与僵尸进程问题**。

2. **Bash 危险命令拦截采用专用事件 + 共享 Channel**
   - **决策**：前端新增 `AgentEvent::PolicyConfirm` 事件用于渲染专用确认 UI（显示危险命令和严重度）。
   - **通道复用**：后端复用现有的 `answer_rx` 通道阻塞等待用户回复（`"yes"` / `"no"`），避免增加新的通道逻辑。

3. **越界检查与环境隔离强化**
   - **路径越界**：在 `file_read`、`file_write` 和 `file_edit` 中统一强制进行基于规范化路径（`canonicalize`）的 Workspace 越界检查。
   - **Bash 越界**：检测命令中的 `cd` 路径，若最终路径越界且未设置 `allow_outside_workspace: true` 则拦截。
   - **环境变量**：`env_clear()` 清空，仅放行 `PATH`, `HOME`, `USER`, `LANG`, `LC_*`, `TERM`, `WORKSPACE_PATH`；同时在默认 allowlist 中加入开发常用变量（`CARGO_HOME`, `GIT_SSH_COMMAND`, `GITHUB_TOKEN`），并读取 `~/.deepseek-code/allowed-env.txt` 支持用户自定义。

4. **Grep/Glob 参数对齐与 ignore 库引入**
   - **Glob**：引入 `ignore` crate（已在桌面端 Cargo.toml 中）以自动解析并尊重 `.gitignore` 规则，支持 `case_insensitive`、`include_hidden`、`max_results`。
   - **Grep**：支持 `include`, `exclude`, `context` (非常实用), `max_count`, `case_insensitive`, `respect_gitignore`。

5. **file_edit 的兼容性**
   - `replace_all: bool` 作为**可选参数**（默认 `false`），防止破坏老版本模型的调用兼容性。

6. **发布策略 (Option A)**
   - 作为一个完整的 `v0.6.0` 版本发布（预计 3 周 / 13-17 工作日），将新工具与 Prompt Injection 防护同步上线，防止 WebFetch 出货时处于裸奔状态。

## 阶段规划

### 阶段 1：快速收益与 UI 接入 (0.5 天)
- **1.1 接入 QuestionCard 到 ToolCallCard**：解决交互式 Q&A 的统一分发。
- **1.2 动态只读工具判定**：在 `agent.rs` 中改用 `tool.is_read_only()` 动态判断分区，移除硬编码。

### 阶段 2：Bash 安全护栏与取消基建 (3 天)
- **2.1 ToolContext 扩展**：加入 `cancel_flag: Arc<AtomicBool>`。
- **2.2 BashTool 重构**：使用 `child.try_wait()` + 轮询实现无泄露的 `timeout_ms` (默认 60s) 和取消机制。
- **2.3 环境变量隔离**：清空并按白名单放行（含常用开发工具环境变量及自定义 allowed-env.txt）。
- **2.4 越界检查**：拦截 `cd` 越界路径。
- **2.5 PolicyConfirm 事件实现**：在 `agent.rs` 执行 bash 前拦截高危命令，通过 `PolicyConfirm` 事件发往前端，并阻塞在 `answer_rx` 上等待用户通过 `ConfirmDialog` 确认。

### 阶段 3：文件与检索工具硬化 (2 天)
- **3.1 路径防越界**：在 `file_read.rs`, `file_write.rs`, `file_edit.rs` 统一加入 canonicalize 越界验证。
- **3.2 原子写入**：`file_write` 和 `file_edit` 改为写 `.tmp` 文件再 `rename` 覆盖。
- **3.3 file_edit 参数可选化**：添加 `replace_all`（可选，默认 false）。
- **3.4 Grep/Glob 升级**：Grep 扩充参数（含 `context` 等），Glob 引入 `ignore` 库读取 `.gitignore`。
- **3.5 question 工具历史守卫**：在重新进入等待前，先在 messages 历史中搜索答案进行复用。

### 阶段 4：新工具集成与网关适配 (7-9 天)
- **4.1 TodoWrite 工具**：SQLite 存储 + `TodoUpdated` 专有事件流式推送。
- **4.2 WebFetch 工具**：`reqwest` 拉取 HTML 并使用 `html2md` 转换，最大限制 8K tokens，增加私有 IP 拒绝（SSRF 防护）及 10s 超时。
- **4.3 WebSearch 工具**：对接 Go 网关 `POST /api/search` 转发 Brave Search API，离线模式 fallback。
- **4.4 SubAgent 工具**：嵌套子 Agent 运行，独立 step 预算与隔离的 `AgentContext`，共享取消信号。

### 阶段 5：Prompt Injection 防护与审计 (2-3 天)
- **5.1 输入隔离包装**：使用 `<<EXTERNAL_UNTRUSTED_CONTENT>>` 标签包裹 WebFetch/WebSearch 结果，并转义内部闭合标签。与 Phase 4.2 的 WebFetch 紧密集成。
- **5.2 危险工具限制**：被不可信源 primed 后，后续的高危 mutating 工具（bash/file_write 等）直接返回 `ToolFailed` 阻止自动执行。
- **5.3 Secret 过滤**：检测工具输出的 API Key/GitHub PAT 等敏感信息并进行 masking 替换。
- **5.4 审计日志**：写入本地 SQLite `safety_events` 表。

### 阶段 6：技术债务清理与文档同步 (1.5 天)
- 编写环境过滤与高危拦截的端到端集成测试，核对 `cargo test` 及 `bun run test`。
- 补齐 `ROADMAP.md`、`.changelog.md`、`CLAUDE.md` 及 `README.md`。

## 验证计划

### 自动化测试命令
- Rust 单元测试：`cargo test -p sidecar-agent`
- Go 单元测试：`cd packages/gateway && go test ./...`
- 前端测试：`bun run test`

### 重点测试覆盖
1. `bash_cancellation_and_timeout` — 验证 try_wait 轮询可以正常杀死超时/被取消的子进程。
2. `path_traversal_prevention` — 验证 `file_read`, `file_write`, `file_edit` 对各种 `..` 和绝对路径越界的拦截。
3. `env_filtering_compatibility` — 确认环境变量清理后，`cargo` 和 `git` 等基础命令仍能正常运行。
4. `untrusted_content_wrapping` — 验证 SSRF 隔离与标签转义。
